# `data/` — pack telemetry, weather, and derived files

This folder is the **persistent state of the cabin install**. The
autonomous loop appends to most files every iteration and pushes
the snapshot to GitHub, so re-cloning the repo gives you the full
history.

## Source-of-truth files (logger-written, append-only)

| File | What it is |
|------|------------|
| **`pack.csv`** | Every BLE poll of both batteries. ~ one row per 10 s. Columns: `ts, state, pack_v, pack_i, pack_p, soc_a, soc_b, v_a, v_b, i_a, i_b, temp_a, temp_b, rem_a, rem_b, delta_v_a, delta_v_b, smoothed_i, smoothed_p, minutes_remaining, name_a, name_b`. Written by `scripts/log.py`. Grows ~ 13 MB / day; this is the canonical source of truth. |
| **`weather.csv`** | Open-Meteo observation every 30 min. Columns: `ts, lat, lon, temperature_c, cloud_cover_pct, shortwave_radiation_wm2, wind_speed_ms, wind_gusts_ms, weather_code, is_day, sunrise_iso, sunset_iso, shortwave_radiation_sum_today_wh_m2, uv_index_max_today`. Written by `scripts/weather.py`. |
| **`pack.log`** | Human-readable progress / BLE-flap notes from `scripts/log.py`. Useful when debugging connection issues. |
| **`launch.log`** | The `Volthium Monitor.app` launcher's chatty log of when services started / failed. |
| **`weather.log`** | Reserved for `scripts/weather.py` output (currently mostly empty). |

## Derived / rolled-up files (loop-regenerated)

| File | What it is |
|------|------------|
| **`daily_summary.csv`** | One row per calendar date with day-total stats (charge/discharge Ah, generator minutes, weather kWh/m², partial flag, etc.). Written by `scripts/daily_summary.py`. Input to `volthium.solar_model.SolarModel.fit_from_daily_summary`. |
| **`calibration_log.csv`** | History of meaningful `SolarModel` coefficient changes. Each row: `ts, coefficient, n_observations, confidence, source, notes`. Written by `scripts/calibration_log.py` (called from `generator_advisor.py`). |
| **`projection_log.csv`** | History of advisor projection snapshots. Each row: `ts, start_soc_pct, projected_sunrise_soc, projected_tomorrow_evening_soc, projected_low_soc, solar_model_coefficient, today_irradiance_kwh_m2, sunrise_iso, source`. Rate-limited to ≥ 25 min apart. Foundation for `scripts/projection_accuracy.py` AND `scripts/low_soc_accuracy.py`. |
| **`solar_onset.csv`** | One row per day capturing the morning solar-onset cascade. Columns: `date, first_zero_iso, first_idle_iso, first_positive_iso, first_net_positive_iso, smoothed_i_at_net_positive, soc_avg_at_net_positive`. The four milestones mark when the pack transitioned from overnight discharge to sustained charging. `soc_avg_at_net_positive` is the day's empirical floor — input to `scripts/low_soc_accuracy.py`. Written by `scripts/solar_onset.py` (idempotent upsert; re-runs enrich the row as more milestones land). |
| **`confidence_log.csv`** | History of advisor confidence-tier transitions. Each row: `ts, base, resolved, lifted, recent_abs_error_pp, recent_n, source`. Only TRANSITIONS in `(base, resolved, lifted)` are logged — stable runs are deduped so the file is a timeline of events, not a stream of duplicates. Written by `scripts/confidence_log.py` (called from `generator_advisor.py` after the accuracy-aware lift is computed). |
| **`voltage_soc_table.csv`** | OCV (open-circuit voltage) → SOC% lookup table extracted from rest-window samples. Written by `scripts/voltage_soc_calibration.py`. Designed to be re-run periodically as the table sharpens with more data. |

## Derived in-memory (no CSV; rendered live)

The dashboard and CLIs compute these views in-memory from the source-of-truth files above:

| View | What it is | Source files |
|------|------------|-------------|
| **Sunrise SOC accuracy** | Per-projection error of `projected_sunrise_soc` vs the actual pack SOC at the matching sunrise (±30 min). | `projection_log.csv` + `pack.csv` |
| **Morning-low SOC accuracy** | Per-projection error of `projected_low_soc` vs the empirical morning low (= `soc_avg_at_net_positive` from `solar_onset.csv`). Surfaced bias finding 2026-05-19: mean -2.97 pp (advisor systematically optimistic about the floor) → led to the sinusoidal-solar fix in `simulate_next_24h`. | `projection_log.csv` + `solar_onset.csv` |
| **Per-horizon accuracy breakdown** | Both accuracy chains bucket records by lead-time (made-at → target gap) in 1h windows. Surfaces systematic lead-time bias. | (same as the two above) |

## Reports

| Path | What it is |
|------|------------|
| **`reports/YYYY-MM-DD.md`** | Markdown day-report generated by `scripts/end_of_day_report.py`. **Six-chain archive** — sections in order: Pack (SOC walk + peaks), Solar harvest, Solar onset (4-milestone cascade for the day), Weather (incl. Open-Meteo forecast revisions), SolarModel state (calibration_log entries dated to the day), Projection accuracy (validated sunrise projections + per-horizon breakdown), Morning-low validation (validated `projected_low_soc` + per-horizon breakdown), Confidence-lift events, Cross-references. Idempotent — re-runs overwrite with the freshest snapshot for the current day; historical days stay frozen, preserving the validation chains' state as they were at end-of-day. |

## Archived backups

| File | What it is |
|------|------------|
| **`pack.csv.v0-1512`** | Snapshot of the pack.csv taken on 2026-05-17 at 15:12 before the schema changed (added `name_a` / `name_b` columns). Preserves the original charge → discharge transition data point. |

## Sizing notes

`pack.csv` grows ~ 13 MB / day at the current 10-s polling cadence
and is the main contributor to repo size growth. Over a month
that's ~ 400 MB. If the repo gets unwieldy we'll revisit log
rotation (e.g. archive older days to `pack-YYYY-MM-DD.csv` files
and keep only the last week unrolled).

## Dashboard surfaces

The live dashboard renders most of this data into a single page;
the log-style files also have their own browseable pages, all
cross-linked:

- `/calibration` — `calibration_log.csv` as a table
- `/projections` — `projection_log.csv` as a table
- `/accuracy` — sunrise SOC validation: per-record table + per-lead-time-horizon breakdown (`projection_log.csv` + `pack.csv`)
- `/low-accuracy` — morning-low SOC validation: per-record table + per-horizon breakdown (`projection_log.csv` + `solar_onset.csv`)
- `/confidence` — `confidence_log.csv` as a table; timeline of accuracy-aware confidence-lift transitions
- `/reports` — index of `reports/*.md`
- `/today-report` — live regeneration of today's day-report
- `/report/YYYY-MM-DD` — specific historical day-report

## Audit-trail topology (how the logs feed each other)

```
pack.csv ──┬──> projection_log.csv ──┐
           │                          ├──> sunrise SOC accuracy view
           │                          │     (and report section)
           │                          │
           └──> solar_onset.csv ──────┴──> morning-low SOC accuracy view
                                           (and report section)
                  ↑                              ↓
                  │                         feeds confidence-lift
              detect on each              decision in advisor
              advisor invocation                 ↓
                                          confidence_log.csv
                                                ↓
                                          confidence-lift events
                                          report section + /confidence

daily_summary.csv ──> SolarModel fit ──> calibration_log.csv
                            ↑                  ↓
                        advisor uses    /calibration view
                        coefficient
```

The advisor (`scripts/generator_advisor.py`) is the orchestrator
that touches every log on each invocation. It reads the SolarModel
fit (from `daily_summary.csv` → updates `calibration_log.csv`),
reads recent `projection_accuracy` records to compute the
confidence-lift, writes a new `projection_log.csv` row (rate-limited),
and writes a new `confidence_log.csv` row only on transitions.
`solar_onset.csv` is updated by the dashboard's `/api/latest.json`
handler on each request (cheap one-day scan over `pack.csv`).
